= Architecture de Visualisation d'Arbres - Solution Complète
:description: Documentation technique de la solution de visualisation d'arbres interactifs
:keywords: arbre de décision, visualisation, architecture, backend, frontend

== Vue d'ensemble

Cette documentation décrit l'architecture complète de la visualisation d'arbres de décision, depuis la génération des données au backend jusqu'à l'affichage interactif dans le tableau blanc.

== Architecture de la solution

=== 🔄 Flux de données complet

[source,text]
----
1. Backend (ML Pipeline)
   ├── Entraînement du modèle
   ├── Extraction de la structure (get_tree_structure)
   └── Stockage dans visualizations['tree_structure']

2. API Gateway
   └── Transmission des données JSON

3. Frontend (Angular)
   ├── Récupération via getTreeData()
   ├── Conversion en nœuds interactifs
   └── Affichage dans tableau blanc
----

== Backend - Génération des données

=== 1. Wrappers de modèles

Les modèles sklearn sont enveloppés dans des classes custom qui exposent `get_tree_structure()`:

[source,python]
----
# ml-pipeline-service/app/ml/algorithms.py

class DecisionTreeWrapper(BaseModelWrapper):
    def get_tree_structure(self, feature_names=None, class_names=None):
        """Extrait la structure d'arbre pour visualisation"""
        tree = self.model.tree_
        
        def build_tree_node(node_id):
            is_leaf = tree.children_left[node_id] == tree.children_right[node_id]
            
            if is_leaf:
                return {
                    "name": class_name or f"Valeur: {value}",
                    "condition": f"n={samples}",
                    "samples": samples,
                    "is_leaf": True,
                    "value": value,
                    "class_name": class_name
                }
            else:
                return {
                    "name": feature_name,
                    "condition": f"≤ {threshold:.3f}",
                    "samples": samples,
                    "is_leaf": False,
                    "feature": feature_name,
                    "threshold": threshold,
                    "children": [left_child, right_child]
                }
        
        return {
            "tree_data": build_tree_node(0),
            "metadata": {
                "max_depth": tree.max_depth,
                "n_nodes": tree.node_count,
                "n_features": tree.n_features
            }
        }
----

----
# ml-pipeline-service/app/ml/evaluation.py

def generate_visualizations(model, X_test, y_test, feature_names=None, 
                          task_type='classification', class_names=None):
    visualizations = {}
    
    # ... autres visualisations ...
    
    # 🌲 NOUVEAU: Structure de l'arbre
    if hasattr(model, 'get_tree_structure'):
        tree_structure = model.get_tree_structure(
            feature_names=feature_names,
            class_names=class_names
        )
        if tree_structure:
            visualizations['tree_structure'] = tree_structure
    
    return visualizations

getTreeData(): any {

=== 2. Conversion en nœuds interactifs

[source,typescript]
----
private convertBackendTreeToWhiteboard(treeData: any): any {
  const nodes: any[] = [];
  const links: any[] = [];
  const nodeConnections: Map<number, number[]> = new Map();
  
  const processNode = (node: any, x: number, y: number, parentIndex: number = -1) => {
    const currentIndex = nodeIndex++;
    
    // Créer le nœud
    const nodeData = {
      id: `node-${currentIndex}`,
      x: x,
      y: y,
      width: 140,
      height: 60,
      isLeaf: node.is_leaf,
      name: node.name,
      condition: node.condition,
      samples: `n=${node.samples}`
    };
    
    nodes.push(nodeData);
    
    // Traiter les enfants récursivement
    if (node.children) {
      node.children.forEach((child, i) => {
        processNode(child, childX, childY, currentIndex);
      });
    }
  };
  
  processNode(treeData, 1000, 200);
  return { nodes, links };
}

== Points clés de l'implémentation

=== Avantages de cette architecture

1. **Séparation des responsabilités**
   - Backend: Extraction pure des données
   - Frontend: Rendu et interaction

2. **Flexibilité**
   - Nœuds HTML individuels vs SVG monolithique
   - Chaque nœud peut être stylé/animé indépendamment

3. **Performance**
   - SVG séparé pour les liens (moins de reflow)
   - Event delegation pour les interactions

4. **Extensibilité**
   - Facile d'ajouter de nouvelles interactions
   - Support pour différents types d'arbres

=== Défis résolus

1. **Synchronisation liens-nœuds**
   - Map de connexions parent-enfant maintenue
   - Recalcul des liens à chaque mouvement

2. **Compatibilité données anciennes**
   - Fallback vers arbre de démonstration
   - Message d'avertissement clair

3. **Transition SVG → HTML**
   - Abandon du parsing SVG existant
   - Construction directe depuis JSON
